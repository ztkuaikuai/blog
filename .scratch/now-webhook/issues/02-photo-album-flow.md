# 02 — 完成图片与 Telegram 相册闭环

**What to build:** 博主可以发布单图或多图 Telegram 相册，博客将其持久化为一条“此刻”；访客通过不暴露 Bot Token 的图片代理查看缩略图和 Viewer.js 预览。相册片段即使乱序或重复到达也能可靠组合，相关编辑与标签撤回保持一致。

**Blocked by:** 01 — 完成文字“此刻”的持久化发布与浏览闭环

**Status:** ready-for-agent

- [ ] 单图消息只持久化 Telegram `file_id` 与必要元数据，不保存图片二进制或含 Token 的下载 URL
- [ ] 分页响应只向浏览器暴露不可猜测的图片记录 ID，不暴露任意 `file_id`、Bot Token 或 Telegram 下载地址
- [ ] 图片代理先查询数据库确认图片属于已发布的“此刻”，再由服务端调用 Telegram `getFile` 并转发内容
- [ ] 图片代理拒绝未知记录和任意 Telegram URL，限制上游超时、响应大小并保留安全 Content-Type
- [ ] 图片响应设置浏览器缓存 1 天、CDN 缓存 7 天和 1 天 stale-while-revalidate，删除后允许最多约 7 天 CDN 残留
- [ ] 单图内容在页面显示缩略图并可通过 Viewer.js 放大预览
- [ ] 相同频道和 `media_group_id` 的消息片段合并成一条多图“此刻”，正文采用带 `#now` 的非空 caption
- [ ] 相册图片按 Telegram 消息 ID 顺序展示，原文链接指向相册第一条消息
- [ ] 相册片段支持乱序和重复到达；任一片段出现标签后整组发布，不会丢失先到达且不带 caption 的图片
- [ ] 未匹配标签的相册保持 pending 且不出现在分页 API；超过 24 小时后在后续 Webhook 请求中惰性清理
- [ ] 相册编辑能够更新对应内容；编辑后移除 `#now` 会删除整个条目和关联图片片段
- [ ] 首版只接受照片和照片相册；视频、GIF、音频、语音、文件、贴纸、投票、位置等内容被安全忽略并返回 200
- [ ] 多图内容在页面横向展示并作为同一 Viewer.js 画廊预览，图片顺序与 Telegram 一致
- [ ] 测试覆盖单图代理授权、Telegram 失败、缓存头、Token 不泄露、相册乱序、去重、排序、pending 发布与清理、编辑和撤回
- [ ] 完整文字与图片闭环通过类型检查、测试和生产构建
