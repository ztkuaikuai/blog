# “此刻”Webhook 持久化与分页展示

## Problem Statement

博主希望继续把 Telegram 频道作为“此刻”的低摩擦发布入口，但当前实现使用 Bot API `getUpdates` 在访客访问时拉取数据。该接口只是最多保留 24 小时的更新队列，不能可靠提供频道历史；Vercel Function 的内存缓存也会随实例生命周期消失。当前图片地址还包含 Bot Token，存在严重的凭证泄露风险。

博主不要求回填 Webhook 启用前的历史，也不要求 Telegram 消息被直接删除后自动撤回，但要求 Webhook 启用后收到的全部“此刻”可靠保存、可分页浏览，并保证服务月成本为零。

## Solution

Telegram 在目标频道发布或编辑消息后，通过 Webhook 将频道更新推送给博客。博客校验 Webhook Secret 和目标频道，筛选带 `#now` 标签的文字、图片及图片相册，并幂等写入 Turso Free。静态生成的“此刻”页面通过 Vercel Function 提供的游标分页 API 读取持久数据。

消息图片只保存 Telegram `file_id`，浏览器通过受约束的图片代理访问。代理从数据库验证图片记录后在服务端向 Telegram 取回文件，并使用 Vercel CDN 缓存；Bot Token 不进入任何客户端响应或请求地址。

## User Stories

1. 作为博主，我想在 Telegram 频道发布带 `#now` 标签的纯文字消息，以便无需提交代码或重新部署即可发布“此刻”。
2. 作为博主，我想使用任意大小写形式的 `#now` 标签，以便 `#NOW`、`#Now` 和 `#now` 都能正常发布。
3. 作为博主，我想发布一张图片和一段文字，以便记录带有视觉内容的近况。
4. 作为博主，我想一次发布多张图片，以便博客将 Telegram 相册展示为一条多图“此刻”。
5. 作为博主，我想让 Webhook 启用后推送成功的全部“此刻”被持久保存，以便内容不会因 Vercel 冷启动或实例回收而消失。
6. 作为博主，我想编辑已发布的 Telegram 频道消息，以便博客同步更新对应的文字与图片信息。
7. 作为博主，我想通过编辑消息并移除 `#now` 标签撤回博客内容，以便无需管理后台即可完成常用撤回操作。
8. 作为博主，我接受直接删除 Telegram 消息不会自动删除博客记录，以便首版无需引入 Telegram 用户账号或额外同步服务。
9. 作为博主，我想通过 Turso 控制台或 CLI 人工删除少量遗留记录，以便无需开发和保护管理后台。
10. 作为博主，我想让 Webhook 重试保持幂等，以便同一 Telegram 更新不会产生重复内容或图片。
11. 作为博主，我想让非目标频道的更新被忽略，以便 Bot 加入其他频道时不会泄露其他频道内容。
12. 作为博主，我想让无关或不受支持的 Telegram 更新被安全忽略，以便它们不会触发 Telegram 的持续重试。
13. 作为博主，我想在本地通过仓库 CLI 注册、查看和删除 Webhook，以便生产部署不会自动抢占或覆盖 Webhook。
14. 作为博主，我想让 Webhook 凭证、Bot Token、频道 ID 与数据库凭证只存在于服务端环境变量中，以便秘密不会进入仓库或客户端包。
15. 作为博主，我想让数据库和 Vercel Functions 保持在免费额度内，以便博客持续运行而不产生费用。
16. 作为博主，我想保留可导出的数据库备份，以便免费服务条款变化时能够迁移内容。
17. 作为访客，我想打开“此刻”页面后立即看到最新内容，以便无需等待服务端访问 Telegram 拉取消息。
18. 作为访客，我想按发布时间倒序浏览“此刻”，以便最新近况总在最前面。
19. 作为访客，我想滚动加载更早的持久内容，以便浏览 Webhook 启用后保存的全部记录。
20. 作为访客，我想在新内容插入期间继续稳定翻页，以便不会因为 offset 漂移看到重复内容或漏项。
21. 作为访客，我想直接看到消息中的图片缩略图，以便快速浏览内容。
22. 作为访客，我想点击图片进行 Viewer.js 预览，以便查看较大版本和多图相册。
23. 作为访客，我想让多图相册保持 Telegram 中的图片顺序，以便内容语义不被打乱。
24. 作为访客，我想保留原文换行和可点击链接，以便文字内容易读且链接可用。
25. 作为访客，我想点击“查看 Telegram 原文”打开对应频道消息，以便查看原始上下文。
26. 作为访客，我不希望图片加载暴露 Telegram Bot Token，以便博客不会因凭证泄露而危及 Bot。
27. 作为访客，我想在没有“此刻”内容时看到“暂无记录”，以便知道页面工作正常但尚未发布内容。
28. 作为访客，我想在首次加载失败时看到明确错误和重试按钮，以便数据库恢复后重新加载。
29. 作为访客，我想在加载更多失败时保留已经显示的内容并单独重试，以便短暂故障不会清空页面或伪装成“已加载全部”。
30. 作为访客，我想在导航中从“文章”后进入“此刻”，以便两个主要内容入口相邻。
31. 作为访客，我想在最近 48 小时有“此刻”发布时看到导航圆点，以便发现近期活跃内容。
32. 作为访客，我接受圆点只表示近期更新而非未读状态，以便博客无需跟踪我的阅读历史。
33. 作为访客，我不希望“此刻”页面自身显示导航圆点，以便当前页面没有多余提醒。

## Implementation Decisions

- “此刻”继续以 Telegram 频道作为唯一发布入口，但以 Webhook 替换访问时 `getUpdates` 拉取。Webhook 与 `getUpdates` 不同时使用。
- Webhook 启用时数据库为空，只保存启用后收到的更新；不回填旧消息。
- 持久化使用 Turso Free（libSQL）。项目不绑定信用卡、不启用 overage；额度耗尽时允许接口降级。数据库需要定期导出备份。
- 博客整体继续静态生成。仅 Telegram Webhook、消息分页 API 和图片代理使用 Vercel Functions 按需运行。
- 数据访问使用 Turso 官方 Serverless SDK和手写 SQL migration。业务端点通过一个集中的数据访问模块读写，不引入 ORM。
- 数据模型包含“此刻条目”和“Telegram 消息片段”两个核心实体。条目负责可见状态、正文、发布时间和分组；片段负责 Telegram 消息 ID、图片 `file_id`、排序和编辑定位。
- 普通消息以频道 ID 与消息 ID 形成唯一分组；相册以频道 ID 与 `media_group_id` 形成唯一分组。Telegram 消息片段也必须在频道内唯一。
- 数据库使用外键级联删除条目对应的消息片段，并为发布时间与稳定 ID 建立倒序分页索引。
- Webhook 仅接受 POST，并限制请求体大小。它必须校验 `X-Telegram-Bot-Api-Secret-Token`，且使用常量时间比较 Secret。
- Webhook 必须校验更新的频道 ID等于配置的目标频道，只接收 `channel_post` 和 `edited_channel_post`。
- Webhook 在一次请求中同步完成数据库事务。提交成功后返回 200；可重试的数据库错误返回 500；合法但无关的更新返回 200。
- 注册 Webhook 时只允许 `channel_post` 和 `edited_channel_post` 更新类型。
- 普通消息只有在文字或 caption 包含大小写不敏感的 `#now` 标签时才发布。展示文字移除标签本身。
- 首版只支持纯文字、照片和照片相册。视频、GIF、音频、语音、文件、贴纸、投票、位置及其他特殊消息不在支持范围内。
- 没有正文也没有图片的空内容不发布。
- Telegram 相册由共享 `media_group_id` 的多条消息组成。相册片段可乱序到达，任一片段包含 `#now` 后整组发布。
- 尚未发现标签的相册片段作为 pending 数据暂存，不向读取 API 返回。超过 24 小时仍未匹配标签的 pending 相册在后续 Webhook 请求中惰性清理，不增加定时任务。
- 相册正文采用非空且包含 `#now` 的 caption；图片按 Telegram 消息 ID 顺序排列；原文链接指向相册第一条消息。
- `edited_channel_post` 通过频道 ID 与 Telegram 消息 ID定位原条目。编辑后仍含标签则更新；编辑后移除标签则删除整个博客条目及其片段。
- Telegram 中直接删除消息不自动同步。首版不提供管理后台或删除 API，低频清理由 Turso 控制台或 CLI 完成。
- 原始文字与 Telegram entities 一并保存。前端保留换行，渲染普通 URL 与 Telegram 标记的文字链接；仅允许 HTTP 与 HTTPS 协议。首版不渲染其他富文本样式。
- 消息分页 API 使用不透明游标。默认每页 20 条，最大 50 条，只返回 published 条目，按发布时间与稳定 ID倒序。
- 分页响应包含消息、下一页游标、是否还有更多以及 48 小时近期更新状态。末页的下一页游标为空。非法或被篡改的游标返回 400。
- 首次请求不带游标。后续游标表示上一页末尾的发布时间与稳定 ID，新消息插入不会改变已经开始的翻页窗口。
- 48 小时圆点表示最新 published 条目是否在当前时间前 48 小时内发布，不表示访客未读。
- Navbar 保留“文章、此刻、标签、友链、关于我”的顺序。“此刻”页面不显示圆点；其他页面使用 sessionStorage 减少同一会话中的重复近期状态请求。
- 数据库只保存 Telegram 图片 `file_id` 和必要元数据，不保存图片二进制或包含 Bot Token 的下载 URL。
- 图片代理只接受博客生成的不可猜测图片记录 ID。代理必须先查询数据库确认记录存在，再使用服务端 Bot Token 调用 Telegram `getFile` 并转发内容；不得接受任意 Telegram URL 或任意 `file_id`。
- 图片代理限制响应大小和上游超时，并保留安全的上游 Content-Type。浏览器缓存 1 天，Vercel CDN 缓存 7 天，并允许 1 天 stale-while-revalidate。
- 条目删除后，已进入 CDN 的图片允许最多约 7 天残留。首版不实现主动 CDN 清除。
- 数据库不可用时读取 API 返回 503。前端不保存消息副本，也不再使用 stale 响应；首次加载和加载更多分别提供重试行为。
- 仓库提供 Webhook set、info 和 delete CLI。脚本从本地未提交的环境文件读取秘密，不打印秘密，也不在 Vercel 部署过程中自动执行。
- 生产上线顺序为：创建 Turso 数据库、执行 migration、配置 Vercel 环境变量、部署生产版本、在本地注册 Webhook、检查 Webhook 状态、发布测试消息。

## Testing Decisions

- 测试外部可观察行为，不断言内部函数调用或 SQL 实现细节。
- 以 HTTP 端点作为最高层测试 seam；使用真实临时 libSQL/SQLite 数据库并执行真实 migration，只 mock Telegram HTTP 边界。
- Webhook 契约测试覆盖 Secret 校验、请求方法、频道过滤、标签大小写、不支持内容、成功响应和可重试失败。
- Webhook 数据测试覆盖重复更新幂等、普通消息、相册乱序、相册去重、图片排序、pending 发布、pending 清理和编辑语义。
- 数据访问与分页通过端点集成测试覆盖稳定游标、默认与最大 limit、无效游标、空结果、分页边界和 48 小时判断。
- 图片代理测试覆盖未知记录、合法图片、Telegram `getFile` 或下载失败、大小与超时限制、缓存响应头、Content-Type，以及响应和 Location 中不出现 Bot Token。
- 前端保留少量 DOM 行为测试，覆盖空状态、首次加载失败重试、加载更多失败保留内容和末页状态；不在首版引入完整浏览器 E2E。
- 现有 Vitest 测试框架继续使用，测试文件保持与源功能目录相对应的镜像结构。
- 实施阶段按 TDD 逐个完成可运行切片：先得到一个针对真实行为的失败测试，再实现到通过，最后重构。

## Out of Scope

- Webhook 启用前的 Telegram 历史回填。
- 使用 Telegram Core API、MTProto、TDLib 或用户账号 Session。
- Telegram 消息直接删除后的自动同步撤回。
- 评论、登录、管理后台和博客侧内容编辑器。
- 视频、GIF、音频、语音、文件、贴纸、投票、位置等非文字/照片内容。
- 富文本的粗体、斜体、删除线、代码块等样式渲染。
- 图片二进制持久化到 Turso、Vercel Blob 或其他对象存储。
- 主动清除 Vercel CDN 中已缓存的图片。
- 访客已读/未读状态、推送通知、WebSocket 或实时前端更新。
- 付费数据库、Vercel Pro、自动 overage 或任何会产生月度费用的服务。
- 完整浏览器 E2E 测试。

## Further Notes

- 架构决策由 ADR-0002 记录，并取代旧的访问时 `getUpdates` 方案。
- Telegram Bot 必须具备接收目标频道 `channel_post` 更新所需的频道权限。
- 免费层额度和条款可能调整；实施和运维时应定期复核 Turso 与 Vercel 的官方限制。
- 预计每日访客少于 100 人，正常调用量远低于 Turso Free 与 Vercel Hobby 当前免费额度。异常流量仍需通过固定分页上限、图片代理约束、CDN 缓存和平台防护控制。
- 研究依据包括 Telegram 历史 API 调研与零成本数据库选型调研。
